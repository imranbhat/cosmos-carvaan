# Implementation Plan: Cosmos Carvaan MVP (Phase 1)

## Overview

Cosmos Carvaan is a used-car marketplace app (inspired by CarSwitch) built with React Native (Expo), Supabase (Postgres + Auth + Storage + Realtime), and Supabase Edge Functions (Deno). This plan covers the 10-week MVP sprint, broken into five two-week phases that are each independently deliverable: Foundation, Core Listings, Sell Flow, Chat and Interactions, and Polish and Launch.

## Assumptions and Constraints

- Single developer (with Claude agent assistance) working full-time
- Supabase Pro plan for production; local Supabase for development
- No dealer features in MVP (Phase 2 per PRD)
- No payment/escrow, inspection booking, or test drive booking in MVP
- i18n deferred to post-MVP; English only for launch
- Target platforms: iOS and Android via Expo EAS Build
- PRD Section 4.1 folder structure is the canonical reference for file paths

## Architecture Summary

```
Frontend:  React Native (Expo) with expo-router, Zustand, React Query
Backend:   Supabase (Postgres + RLS + Auth + Storage + Realtime + Edge Functions)
Shared:    packages/shared/ for types, validators, constants
Monorepo:  Flat structure (not Turborepo for MVP simplicity)
```

- `supabase/migrations/007_indexes.sql`

**Actions:**
1. Create `profiles` table with trigger to auto-create on auth.users insert
2. Create `car_makes`, `car_models`, `car_variants` tables with foreign keys
3. Create `listings` and `listing_photos` tables with status enum
4. Create `saved_cars`, `saved_searches`, `offers`, `test_drives` tables
5. Create `conversations` and `messages` tables with message type enum
6. Write RLS policies per PRD Section 4.4 plus:
   - Profiles: users can read all, update only own
   - Listing photos: public read for active listings, seller manages own
   - Saved cars: user manages own only
   - Conversations: participants only (buyer_id or seller_id matches auth.uid())
   - Messages: conversation participants only, sender can insert
7. Create indexes on: `listings(status, created_at)`, `listings(make_id, model_id)`, `listings(city)`, `listings(price)`, `messages(conversation_id, created_at)`, `saved_cars(user_id)`

9. Seed `car_makes` (top 30 brands: Toyota, Honda, Nissan, BMW, Mercedes, etc.) and `car_models` (top 5-10 models per make) and `car_variants`

**Dependencies:** Step 1.1 (Supabase initialized)
**Agent:** **architect** agent for schema review; use `postgres-patterns` skill for migration best practices
**Risk:** High -- RLS policies are security-critical and easy to get wrong. Mitigation: write integration tests that verify RLS by querying as different user roles; use **security-reviewer** agent after writing policies.

**Acceptance Criteria:**
- [ ] `npx supabase db reset` runs all migrations and seed without errors
- [ ] RLS test: anonymous user can SELECT active listings but not INSERT
- [ ] RLS test: authenticated user can only UPDATE their own profile

**Actions:**

2. Build `api.ts` with functions: `fetchListings(filters, sort, page)`, `fetchListingById(id)`, `fetchSimilarListings(id)`, `fetchMakes()`, `fetchModels(makeId)`, `fetchVariants(modelId)`.

5. `fetchSimilarListings` queries same make or similar price range, excludes current listing, limits to 10.
6. Build React Query hooks wrapping each API function with proper query keys for cache invalidation.

**Risk:** Medium -- Complex filter combinations could produce slow queries. Mitigation: composite indexes in migration 007, use EXPLAIN ANALYZE during development.
